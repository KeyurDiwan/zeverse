1. Executive summary

Zeverse is a project-agnostic control plane: an Express API and workflow runner (server/), a Vite + React dashboard (ui/), and an optional Slack Bolt bot (slack-bot/) that share one pipeline. Business logic for what to automate lives in each target repository under .archon/workflows/ and .archon/commands/ — not inside the Zeverse repo itself.

Runs create an isolated git workspace (usually ephemeral), execute YAML-defined steps (LLM, shell, patches, integrations), optionally pause for human approval or Slack thread replies, then push a branch and open a pull request when branch isolation is enabled.

Mental model: Zeverse = registry + orchestration + surfaces. Each app repo = workflows + conventions + optional per-repo harness routing. See section 3 for how a single user request moves through the stack, and section 4 for why conventions live beside application code.

2. High-level architecture

Users and automations reach the API on port 3100 (configurable). The server loads workflow definitions from the registered repo’s checkout or cache, acquires a session (clone path + branch), runs step executors, and writes state under state/<repoId>/runs/. External systems: git remote (GitHub), LLM (CloudVerse / config in config/archon.yaml), optional Google Docs, Freshrelease, Confluence — depending on workflow steps and env vars.

3. End-to-end request flow

A “request” is anything that eventually triggers work on a registered repo: a click in the UI, a Slack message or button action, or an HTTP client calling the API. The path splits into harness-mediated flows (route → confirm → execute, typical for Slack) and direct runs (POST /api/run-workflow when the caller already knows repoId and workflow).

3.1 Layers of responsibility

LayerRole
SurfaceCollects intent (prompt, repo hint, branch, Slack channel/user metadata).
APIValidates JSON bodies, applies policy, resolves repos.json, loads YAML from workflow cache.
Harness (optional)Selects workflow via keywords, per-repo harness.yaml, or hub LLM; returns proposal vs answer vs clarify.
RunnerstartRun: allocates run id, persists state, acquires git session (clone/fetch), executes steps in order.
ExecutorsPer-step backends: LLM chat, subprocess shell, patch/apply/edit, Google Docs, Freshrelease, child workflows, gates.
PersistenceAppend logs; NDJSON events; optional audit line on harness execute; git push + PR when isolation demands it.

3.2 Harness-mediated flow (Slack / universal routing)

  1. Ingress — Slack Bolt (or another client) sends POST /api/harness/route with at least { "prompt": "..." }. Optional: repoId, threadContext (prior turns), surface.
  2. Resolution — Server resolves repo (explicit id → else LLM inference over registry → else clarify). Empty workflow list yields an informational answer.
  3. Routing — Keyword match → immediate proposal. Else dry-run repo harness workflow (catalog + route steps). Else hub builds workflow catalog + optional repo rules text and asks the LLM for structured JSON.
  4. Response contract — proposal includes workflow, merged inputs, confidence, reason. Low confidence or unknown workflow → answer (chat-only). Missing repo → clarify.
  5. Human confirm — Slack bot renders Run / Pick another / Cancel. No mutation of the remote repo happens until confirmation.
  6. Execute — On confirm: POST /api/harness/execute with repoId, workflow, inputs/prompt, optional slackUser, channel, surface, baseBranch. Policy check → validate required workflow inputs → startRun → audit log line.
  7. Observe — Clients poll /api/runs/:id, /api/logs/:id, /api/runs/:id/events. Slack maps milestone step ids to thread posts when configured.

3.3 Direct run flow (UI / automation)

When the caller already chose repo + workflow + inputs (for example the dashboard run form or CI): POST /api/run-workflow with repoId, workflow, optional prompt/inputs/baseBranch. The server merges requirement from prompt, autofills known inputs from URL patterns (Freshrelease, Google Doc, GitHub PR, etc.) where declared, then calls startRun immediately — no harness proposal step.

3.4 Inside startRun (conceptual)

  1. Create runId and initial RunState (all steps queued).
  2. Session — Acquire ephemeral clone or managed workspace; create run branch when using branch isolation.
  3. Template context — Bind inputs.*; each finished step exposes steps.<id>.output to later steps.
  4. Step loop — For each step: evaluate when:, resolve retries/loops, dispatch executor, append logs + emit events.
  5. Blocking steps — approval and wait-thread-reply pause until POST approve/reject/thread-reply.
  6. Terminal — Evaluate workflow gates; optionally run onGateFail; commit/push/open PR if applicable; cleanup ephemeral dir.
Compare: direct /api/run-workflow skips route + confirmation — suitable when automation already chose workflow and inputs. Slack almost always uses the harness so users confirm destructive or repo-wide actions.

4. Rules, skills & repo-local context

Zeverse is deliberately repository-agnostic: one hub process may orchestrate dozens of unrelated codebases (React apps, services, libraries). The control plane cannot embed framework-specific lint rules, naming conventions, security constraints, or testing commands for each product — those evolve with the application and belong next to that application’s source.

4.1 Why add rules / skills in each target repo?

Single source of truth for humans and automation
Markdown “rules” document stack choices (framework versions, bundler, CSS strategy), coding conventions, how tests are run, and domain vocabulary. The same files can align Cursor / IDE assistants with Zeverse behaviour so engineers do not maintain two conflicting playbooks.
Better routing and safer automation
Server-side harness fallback routing attaches aggregated repo rules text to the router LLM system prompt (loadRepoRules in server/src/workflows.ts), improving workflow selection and reducing proposals that contradict documented conventions.
Faster onboarding after import
Running bootstrap-rules fingerprints the tree (layouts, configs, CI hints) and opens a PR that seeds .archon/rules/*.md — teams start from structured commentary instead of blank prompts.
Scoped blast radius
When rules live in-repo and ship via PR, reviewers see convention changes alongside code ownership boundaries; the hub never needs redeploying to adopt them.

4.2 Where rules are loaded from (implementation)

The server merges text from the workflow checkout/cache in this order (see loadRepoRules):

  1. .archon/rules/*.md — Primary Archon/Zeverse convention files (often produced or edited after bootstrap).
  2. .cursorrules — Single-file Cursor rule bundle at repo root (truncated if extremely large).
  3. .cursor/rules/*.md — Cursor-style modular rules.

Files are concatenated with section headers so the router LLM can attribute snippets. Total combined length is capped to protect token budgets.

4.3 Bootstrap workflow (bootstrap-rules)

Trigger via UI “+ Rules”, Slack button after add-repo, or POST /api/repos/:id/bootstrap-rules. A dedicated workflow clones the repo, fingerprints structure, asks an LLM to draft Markdown files under .archon/rules/ (for example tech stack, conventions, testing), applies them into the workspace, then follows the normal branch isolation path — branch name pattern archon/bootstrap-rules/<runId> — resulting in a GitHub PR teams review like any other change.

4.4 Relationship to workflow YAML prompts

Individual workflow steps define their own LLM prompts in YAML; authors often embed repo-specific instructions there or rely on shell steps to read files from the workspace. Keeping durable conventions in .archon/rules/ avoids duplicating long boilerplate across every workflow while remaining merge-reviewed like application docs.

4.5 “Skills” in naming

Product docs sometimes pair “rules & skills” with bootstrap — practically this means curated markdown capability/context packs plus executable workflows/commands under .archon/ shipped by that repo. Zeverse executes whatever workflows exist in .archon/workflows/; naming aligns with Cursor terminology for familiarity across tooling.

5. Integrating another repository

5.1 Register the remote

Add the git origin to Zeverse’s registry (stored in repos.json at the project root by default). Surfaces:

  • UI — Import flow in the dashboard.
  • HTTP — POST /api/repos with URL and optional name.
  • Slack — add-repo (gated by ARCHON_REPO_ADMIN_USER_IDS and channel policy).

Zeverse stores the remote URL; it does not require a permanent local clone at import time. Each run typically creates an ephemeral clone under state/<repoId>/runs/<runId>/work/, or reuses a managed workspace at repos/<id>/ when the workflow sets keepWorkspace: true (serializes runs per repo, faster for huge trees).

5.5 Onboarding checklist

  1. Import repo → verify row appears in repos.json with correct defaultBranch.
  2. Add at least one workflow YAML under .archon/workflows/; refresh cache via UI or POST /api/repos/:id/refresh-workflows when iterating locally.
  3. Optional: add harness.yaml with catalog/route for custom routing prompts.
  4. Optional: run bootstrap-rules to seed .archon/rules/, or maintain .cursorrules / .cursor/rules/.
  5. Configure secrets at hub root (GITHUB_TOKEN, integration tokens) so pushes and PR creation succeed.
  6. Smoke-test: direct run via UI or /api/run-workflow, then Slack harness proposal → execute.

5.2 Add workflows and optional harness

In the target repo, create YAML files under .archon/workflows/. Example shape:

name: dev
description: Plan, implement, validate, review, PR.
inputs:
  - id: requirement
    label: Feature requirement
    required: true
steps:
  - id: plan
    kind: llm
    prompt: |
      Break this requirement into ordered tasks:
      {{inputs.requirement}}

Optionally add .archon/workflows/harness.yaml with steps such as catalog and route. When present, POST /api/harness/route dry-runs those steps via runSingleStep to pick a workflow. If missing or parsing fails, the server falls back to keyword routing and then server-side LLM routing using the workflow list and .archon/rules context.

5.3 Bootstrap rules (optional)

POST /api/repos/:id/bootstrap-rules fingerprints the codebase and opens a PR adding .archon/rules/*.md so future runs get consistent repo context.

5.4 Branch targeting

Override the base branch per run: API body baseBranch, UI branch field, or Slack branch=<name>. Defaults to the repo’s defaultBranch in repos.json.

6. Workflow model and step kinds

6.1 YAML shape and templating

Each workflow file declares name (referenced when starting runs and routing), human-readable description, ordered inputs (each with id, label, optional required), and steps. Before each step runs, prompts and shell commands are rendered through the template engine: {{inputs.<id>}} expands user-supplied strings; {{steps.<stepId>.output}} expands prior step stdout/text results, enabling pipelines such as “lint → feed failures back into fix loop”.

Optional fields on steps include when: (expression — empty/false/no/0 skips), retries / retryBackoffMs for flaky commands, loopUntil plus maxIterations for convergence loops, and continueOnError where supported so validation failures do not abort early unless gates demand success.

7. Run lifecycle

A run receives a runId, is tracked in memory and on disk, moves through queued steps, streams logs, and emits NDJSON events to state/<repoId>/runs/<runId>.events.ndjson for UIs and the Slack bot to poll (GET /api/runs/:id/events). Steps may pause for approval or wait-thread-reply. After all steps complete, workflow gates are evaluated; branch isolation triggers commit, push, and PR creation when configured. Ephemeral workspaces are removed after completion.

7.1 State and logs on disk

  • Per-run folder — Under state/<repoId>/runs/: serialized run state JSON, textual logs for tail endpoints, NDJSON event stream for milestone UX.
  • In-memory active runs — Recent runs stay addressable while the server process holds them; persisted copies survive restarts when written.
  • Working tree — Ephemeral clones avoid cross-run contamination; managed workspaces reuse one directory with locking.

7.2 Git outcomes

When isolation: branch, successful edits commit on archon/<workflow>/<runId> (pattern may vary by implementation details), push to origin, then open or update a pull request via gh or GitHub REST when tokens permit. Empty runs may still produce empty commits depending on workflow — callers rely on logs for truth.

8. Harness and Slack pipeline

Slack, the UI, and direct API calls converge on two harness endpoints: route (propose or answer) and execute (start a real run after confirmation). The Slack bot always shows confirmation before a write-capable workflow runs.

8.1 POST /api/harness/route

Resolution order (see server/src/routes/harness.ts):

  1. Repo — Use repoId from the request if provided; otherwise infer via LLM over the registry (or clarify if none/ambiguous).
  2. Keyword shortcut — If matchWorkflowKeyword matches, return type: proposal with high confidence.
  3. Per-repo harness — If a workflow named harness exists and has a route step, optionally run catalog, then route via runSingleStep. Parse JSON from the route output; if workflow is known and confidence ≥ 0.6, return proposal; else answer. On failure, fall through.
  4. Server LLM fallback — Build a catalog of non-harness workflows, merge repo rules text (loadRepoRules: .archon/rules, .cursorrules, .cursor/rules) when non-empty, and ask the LLM for JSON: workflow, inputs, alternatives, confidence, reason. Same 0.6 threshold; unknown workflow or low confidence yields answer with explanatory text.

Response type is one of:

  • proposal — Suggests workflow, inputs, alternatives, confidence, reason.
  • answer — Direct LLM reply when routing is uncertain or the user should not get an auto-run.
  • clarify — Missing repoId or invalid repo; includes question and missing.

8.2 POST /api/harness/execute

Validates repoId and workflow, applies policy (assertAllowed), merges inputs (including requirement from prompt), extracts Freshrelease URLs into frUrl when needed, checks required workflow inputs, then calls startRun and appends one line to state/audit.log.

8.3 Slack surfaces (expanded)

  • Slash commands — /archon-harness is the universal router; /archon-dev, /archon-prd, /archon-add-repo specialize common flows.
  • @mentions — Invite the bot to a channel, then mention it; behaviour matches harness routing + confirmation.
  • DMs — Same behaviour without requiring a mention.
  • Thread context — Follow-ups send earlier turns as threadContext so “now fix it” resolves against prior diagnosis.
  • PRD threads — Commands such as “answer this” vs “update the PRD doc” sync discussion back to Google Docs (see repository README).

8.4 Example payloads (conceptual)

POST /api/harness/route
{ "prompt": "fix flaky login test on checkout", "repoId": "my-app" }

// Route — Slack supplies thread history

  "threadContext": "assistant: … user: …", "surface": "slack" }

}

9. Policy, audit, and observability

9.1 Policy enforcement

Operators may tighten blast radius without code changes by uncommenting / editing policy: inside config/archon.yaml: restrict which repoId values, workflow names, or Slack channel ids may invoke runs. Defaults commonly allow all (["*"]). Checks run inside /api/harness/execute today (direct /api/run-workflow does not call the same gate — secure those paths separately if exposed). Violations surface as HTTP 403 with a machine-readable reason.

9.2 Audit trail

Each successful harness execution appends one JSON object per line to state/audit.log, capturing who triggered work (Slack user id when known), channel, repo, workflow name, run id, and surface — enabling downstream SIEM ingestion or quarterly reviews. Secrets never appear in these records by design.

9.3 Observability streams

  • Logs — Human-readable append-only logs per run for /api/logs/:id tailing.
  • Events — Structured NDJSON (step_started, step_finished, awaiting_approval, gates_failed, run_finished, …) for deterministic automation.
  • Milestones — Optional runner.milestone_steps list maps significant step ids to Slack bot announcements.
